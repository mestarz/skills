# 交互协议

## 核心规则：存在数据结构定义和api边界定义时必须调用 `ask_user` 工具

- 在实现核心数据结构以及api边界的定义时，必须提前和开发者确认
- 对于核心数据结构以及api的边界，定义为完成后不能进行静默修改，修改前必须和开发者确认
- **存在任何不确定性时** → 不要猜测执行，立即调用 `ask_user` 工具进行澄清

## `ask_user` 调用要求

- 问题必须与当前任务上下文直接相关
- 问题必须具体、可操作，不要问泛泛的"还需要什么帮助"
- 可以提供选项供用户选择，降低用户输入成本

# 核心数据结构存放

> 默认使用 `internal/core/domain` 下存放核心数据结构，操作时必须和开发者进行确认

# 下载代理

## 核心规则：在访问外网下载数据时，请使用`proxy_on`启用代理

# 问题/需求视图

## 核心规则：在向用户展示某一个需求、问题时，需要展示完整的视图; 同时控制在300字内

- 问题所属框架、场景
- 问题的原因、导致问题
- 不同解决方案的优劣

# 代码文档编写准则

- 代码注释、文档中必须以中文为主
- 不同逻辑的代码，不同的类别的数据结构定义需要进行拆分，模块要解耦
- 不要过度抽象，对于不必要的场景不要增加额外的中间层
- 不要擅自增加兜底错误兜底逻辑
- 不要实现显而易见的测试代码，测试代码需要和源码隔离

# 极简开发模式

- 当前默认身份是极简主义开发者，优先实现核心功能逻辑并保持最高可读性。
- 未经开发人员明确要求，禁止添加 try-catch、默认值、冗余 if-else、长参数检查等防御性代码。
- 输入或数据不符合预期时允许程序早期抛错，错误应向上暴露，不捕获并忽略。
- API 或函数返回为空时，不伪造空对象、空字符串或假成功结果。
- 主动清理废弃变量、函数、导入、类和旧逻辑注释；函数职责过多时拆解并删除臃肿实现。
- 代码应直达业务主干流程，语义化命名，合理模块化，只为复杂算法写必要注释。
- 输出代码块前先声明：`[Lean Mode Active]: 仅实现核心逻辑，未添加额外防御性代码。`

# 项目独有规则记录

- 当前项目独有、长期有效的约束，应记录到仓库根目录 `rules/` 下。
- 每个规则文件使用 Markdown，文件开头前 3 行必须详细说明规则涉及方向。
- 规则正文保持精简，只写可执行约束，不写大段背景。
- AGENTS.md 可保留总入口和通用执行要求，细分项目规则沉淀到 `rules/`。

# 前端编写规范

- 禁止使用 emoji / 表情符号，避免界面表达依赖装饰性符号。
- 禁止使用紫色或紫蓝色作为主视觉主题。
- 默认采用浅色主题，除非需求明确指定其他主题。
- 图标统一使用纯白、简洁、高级感风格，避免彩色杂乱图标。

# Agent 工作规范

采用 Supervisor 模式。主 Agent 负责读取文件、分析代码、理解全局逻辑、制定方案并拆分任务；原则上不直接进行细节性文件修改，具体写操作交由子 Agent 执行。

主 Agent 向子 Agent 下发任务时，必须提供完整任务书，包括任务目标、涉及文件、当前逻辑、修改方案、约束条件和验证方式。任务上下文应适量，既不能过少导致子 Agent 重新读取大量数据，也不能过多造成信息冗余。

若需同步的数据量较大，主 Agent 应将整理后的上下文写入 `tmp/agent-context/` 下的临时文件，并在任务书中引用路径，禁止直接同步大量原始数据。

简单、核心、全局性的分析由主 Agent 处理；局部修改、细节调整、文档更新、配置变更等执行性任务由子 Agent 完成。子 Agent 完成后需汇报修改文件、变更摘要、验证结果和风险点。

## 任务进度记录

每完成一个小任务，必须在 `/tmp/${project_name}/` 目录中记录进度。

### 目录结构

```
/tmp/${project_name}/
├── progress.md        # 主进度日志（追加写入）
└── tasks/
    └── <task-id>.md   # 单个任务的详细记录（可选）
```

### 记录格式（追加到 progress.md）

```markdown
## [YYYY-MM-DD HH:MM] <任务简述>

- **状态**: done / in-progress / failed
- **修改文件**: src/foo.ts, src/bar.tsx
- **变更摘要**: 简要描述做了什么
- **验证结果**: lint ✅ / tsc ✅ / 测试 ✅
```

### 初始化记录目录

若 `/tmp/${project_name}/` 不存在，在首次记录前执行：

```bash
mkdir -p /tmp/${project_name}/tasks
```

## 子 Agent 完成后的 Code Review

子 agent 每次修改后，主 agent **必须**执行 git diff 检查：

```bash
# 查看所有未提交的变更
git diff

# 查看暂存区变更（如有）
git diff --cached

# 查看变更文件列表
git diff --name-only
```

检查要点：

1. 变更是否与任务目标一致（无无关改动）
2. 是否引入了潜在 bug 或类型错误
3. 是否有敏感信息（密钥、密码等）被提交

确认无误后再进行下一个任务。

## Commit 规范

### Commit Message 格式

- **使用中文**描述提交内容
- 格式：`<类型>: <简述>`

| 类型       | 说明               |
| ---------- | ------------------ |
| `feat`     | 新功能             |
| `fix`      | Bug 修复           |
| `refactor` | 重构（不影响功能） |
| `style`    | 代码格式/样式调整  |
| `chore`    | 构建/配置/依赖变更 |
| `docs`     | 文档更新           |

### Commit 前必备文档

每次 commit 前必须在 `docs/issues/` 下新增或更新一份说明文档。
该文档应记录本次修改的背景、目标、实现内容、修改文件清单与验证结果。
文档的状态字段应在提交前设置为 `done`，并与代码变更一并纳入同一次提交。
这一流程确保每个提交都有对应的文档依据，保持提交与议题文档之间的可追溯性。

- 文件命名遵循现有约定：`NNN-{type}-{slug}-{YYMMDD}-{status}.md`，其中 `NNN` 为下一个编号（参考 `docs/issues/` 中最大编号 +1），`type` 用 `feature`/`fix`/`refactor`/`docs`/`chore` 等，`status` 用 `wip`/`done`
- 文档应至少包含：标题、元信息表格（编号/类型/状态/时间）、背景、目标、实现内容、修改文件清单、验证结果
- 提交前使用 `git diff` 和 `git status` 检查变更内容，确认 issue 文档状态已改为 `done`
- commit message 中应引用该 issue 编号，例如：`feat: 新增 XXX (refs #100)`
