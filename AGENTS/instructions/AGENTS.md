# 交互协议

## 核心规则：每次回复结束前必须调用 `ask_user` 工具

> 特殊除外场景，当进行plan和build模式切换时，需要调用对应的工具

这是不可跳过的强制协议。在你的每一轮回复中，你必须执行以下操作之一：

1. **完成用户请求后** → 立即调用 `ask_user` 工具，提出与当前上下文相关的后续问题
2. **存在任何不确定性时** → 不要猜测执行，立即调用 `ask_user` 工具进行澄清

## 禁止行为

- **禁止在不调用 `ask_user` 的情况下结束回复**
- **禁止使用终结性表达**（如"希望对你有帮助"、"如有问题随时提问"等）
- **禁止猜测用户意图** — 不确定就用 `ask_user` 询问

## `ask_user` 调用要求

- 问题必须与当前任务上下文直接相关
- 问题必须具体、可操作，不要问泛泛的"还需要什么帮助"
- 可以提供选项供用户选择，降低用户输入成本

# Agent 工作规范
> 约束 AI agent 在此项目中的工作方式。

## 角色分工：Supervisor 模式

### 主 Agent（Supervisor）职责
- **只允许**：读取文件、分析代码、启动子 agent
- **禁止**：直接修改任何文件（源码、配置、文档均不允许）
- 所有写操作（新建/编辑/删除文件）必须委托给子 agent 执行

### 子 Agent 职责
- 负责具体的代码修改、文件创建、重构等写操作
- 修改完成后主动汇报变更摘要

## 任务进度记录

每完成一个小任务，必须在 `/tmp/${project_name}/` 目录中记录进度。

### 目录结构
```
/tmp/${project_name}/
├── progress.md        # 主进度日志（追加写入）
└── tasks/
    └── <task-id>.md   # 单个任务的详细记录（可选）
```

### 记录格式（追加到 progress.md）

```markdown
## [YYYY-MM-DD HH:MM] <任务简述>

- **状态**: done / in-progress / failed
- **修改文件**: src/foo.ts, src/bar.tsx
- **变更摘要**: 简要描述做了什么
- **验证结果**: lint ✅ / tsc ✅ / 测试 ✅
```

### 初始化记录目录

若 `/tmp/${project_name}/` 不存在，在首次记录前执行：
```bash
mkdir -p /tmp/${project_name}/tasks
```

## 子 Agent 完成后的 Code Review

子 agent 每次修改后，主 agent **必须**执行 git diff 检查：

```bash
# 查看所有未提交的变更
git diff

# 查看暂存区变更（如有）
git diff --cached

# 查看变更文件列表
git diff --name-only
```

检查要点：
1. 变更是否与任务目标一致（无无关改动）
2. 是否引入了潜在 bug 或类型错误
3. 是否有敏感信息（密钥、密码等）被提交

确认无误后再进行下一个任务。

## Commit 规范

### Commit Message 格式
- **使用中文**描述提交内容
- 格式：`<类型>: <简述>`

| 类型 | 说明 |
|------|------|
| `feat` | 新功能 |
| `fix` | Bug 修复 |
| `refactor` | 重构（不影响功能） |
| `style` | 代码格式/样式调整 |
| `chore` | 构建/配置/依赖变更 |
| `docs` | 文档更新 |
